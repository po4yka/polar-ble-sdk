# KMP Shared Artifact Consumption

This document records how the `:shared` module is consumed as migration slices intentionally delegate Android or iOS behavior to shared KMP code. Android and iOS modules may depend on shared code only when a behavior slice adds failing platform characterization tests, common tests, and facade compatibility checks for the delegated behavior.

## Current Artifact Shape

The shared module lives at `sources/Android/android-communications/shared` and is included as Gradle project `:shared` from `sources/Android/android-communications/settings.gradle`. It exposes production common code for migrated Android slices, common golden-vector test loading, a JVM test target for executable `commonTest`, an AGP 9 Android KMP library target, and Apple targets `iosX64`, `iosArm64`, `iosSimulatorArm64`, `watchosX64`, `watchosArm64`, and `watchosSimulatorArm64`.

## Android Consumption Contract

Android production code should consume shared code through a Gradle project dependency only in the migration slice that delegates behavior: `implementation project(':shared')` in `sources/Android/android-communications/library/build.gradle`. Before adding that dependency, the slice must prove the current Android behavior with existing or new Android characterization tests, add or reuse shared golden vectors, add the matching KMP common tests, and keep `GoldenVectorMigrationPolicyTest` green. The Android artifact smoke gate for target wiring is `./gradlew :shared:bundleAndroidMainAar :shared:compileAndroidMain --no-daemon` from `sources/Android/android-communications`.

Release AAR consumption uses a documented two-AAR compatibility model during the compatibility phase: `polar-ble-sdk.aar` for the Android facade and `polar-ble-sdk-shared.aar` for the KMP implementation dependency. `sources/Android/android-communications/scripts/build_sdk.sh` copies both files into `SdkBuild`, and `scripts/verify_android_example_aar_consumption.sh` builds the SDK artifacts, copies them into `examples/example-android/polar-sensor-data-collector/polarBleSdk`, and runs the example AAR metadata, duplicate-class, and dex-merge gates with `-PlocalSdk=true` so the example proves the local AAR pair is consumable without coupling release packaging validation to the example app's Hilt/KSP source-generation toolchain. The shared module also defines Gradle module metadata through `maven-publish` for Maven-style consumers, with `polar-ble-sdk-shared` as the root KMP publication and target publications such as `polar-ble-sdk-shared-android`; `scripts/verify_android_shared_maven_publication.sh` publishes only to a temporary local repository and asserts the generated `.module` metadata plus Android shared AAR exist. CI remains artifact-only and does not publish to Maven, JitPack, CocoaPods, SwiftPM, or any external host without a separate secrets/release-policy plan. Direct file-based example consumption must keep both AARs until the SDK AAR is deliberately repackaged as a fat artifact or the shared code is no longer referenced from public Android facade classes.

## iOS Consumption Contract

iOS production code consumes shared code through the generated static framework `PolarBleSdkShared.framework`, keeping the existing Swift `PolarBleSdk` and `iOSCommunications` public APIs as the current Swift facade compatibility boundary. `sources/iOS/ios-communications/scripts/build_kmp_ios_framework.sh` selects `iosArm64`, `iosX64`, or `iosSimulatorArm64` from the active Xcode platform and architecture, runs the matching Gradle framework link task, and copies the generated framework under ignored `sources/iOS/ios-communications/Generated/PolarBleSdkShared/$(PLATFORM_NAME)`. The Xcode project and CocoaPods podspec add that path to `FRAMEWORK_SEARCH_PATHS`, link `PolarBleSdkShared`, and run the script before Swift compilation for the iOS framework targets. The validated CocoaPods/Xcode iOS `PolarBleSdk` packaging surface defines `POLAR_KMP_SHARED_REQUIRED`; `PolarSharedFrameworkLinkGuard` makes that surface fail at compile time if `PolarBleSdkShared` is not importable, while Swift Package Manager and watchOS builds do not define the condition and remain source-compatible through the existing Swift fallback until a later release adds a checked-in or downloaded binary artifact packaging strategy. The first iOS production consumption slices delegate Device ID/UUID, advertisement local-name model extraction and manufacturer HR payload extraction, selected time/date utilities, selected-config product capability lookup, compatible type utility conversions, PMD active-measurement status-bit mapping, PMD measurement-type ID lookup, PMD recording-type request-bit encoding, PMD selected-setting serialization and setting-type lookup, PMD secret strategy serialization/from-byte mapping, MAG calibration-status lookup, disk-space byte-total mapping, D2H notification ID-to-type mapping, training-session reference aggregation, exercise sport-profile lookup, SD-log trigger and magnetometer-frequency enum mapping, SPo2 enum boundary mapping, REST decoded model projection, offline-recording filename-to-measurement mapping and PMDFILES.TXT V2 grouping, skin-temperature enum mapping, backup default path policy, firmware utility version/file-priority helpers plus package write-order planning, watch-face complication ID lookup, sleep wake-state/rating enum mapping including proto-read sleep rating projection, automatic-sample HR/PPi trigger and PPi status mapping, automatic-sample PPi status helper mapping, activity-class mapping, daily-balance feedback mapping, training-readiness mapping, user-device-settings enum/value read/write plus public device-location string helper mapping, first-time-use physical-config enum mapping, stored-data cleanup filter/prune decisions, and stored-data type public helper mapping through `PolarIosSharedBridge`; future slices should add the same XCTest, common-test, and facade-compatibility evidence before removing additional Swift logic. The Apple artifact smoke gate for target wiring is `./gradlew :shared:linkDebugFrameworkIosX64 :shared:compileKotlinIosX64 --no-daemon` from `sources/Android/android-communications`, followed by the iOS XCTest command in `KmpValidationCommands.md`.

CocoaPods is the supported iOS shared-consumption package path for this phase. `PolarBleSdk.podspec` preserves `sources/iOS/ios-communications/scripts/build_kmp_ios_framework.sh`, links `PolarBleSdkShared`, and is verified by `pod lib lint PolarBleSdk.podspec --allow-warnings` after `pod install --project-directory=sources/iOS/ios-communications`.

Swift Package Manager and watchOS are fallback-only on a clean checkout and may consume shared KMP only through an explicit `PolarBleSdkShared.xcframework` binary artifact. `sources/iOS/ios-communications/scripts/package_kmp_xcframework.sh` is the local-output packaging path: it builds the iOS and watchOS Kotlin/Native frameworks, creates `sources/iOS/ios-communications/Generated/PolarBleSdkSharedXCFramework/PolarBleSdkShared.xcframework`, and can optionally zip the artifact and print a SwiftPM `checksum`. `Package.swift` declares a conditional local `binaryTarget` named `PolarBleSdkShared` only when that generated XCFramework path exists, so `swift package describe` remains valid on clean checkouts while local package builds can verify the binary target after running the packaging script. Release distribution must use the same artifact shape with either a checked-in binary path or a remote `binaryTarget(url:checksum:)` entry produced from the zipped XCFramework. Do not claim SwiftPM/watchOS shared consumption from `swift package describe` alone; the claim requires a built or downloaded `PolarBleSdkShared.xcframework`, package resolution that includes the `PolarBleSdkShared` binary target, and a SwiftPM iOS/watchOS build or equivalent package integration gate. Until that validation exists for a release artifact, the fallback Swift code remains the compatibility path.

## Rollback Paths

Every shared-module adoption step must keep a rollback path that can be applied without changing public SDK APIs. For Android utility/model/parser slices, rollback is to remove the `Polar*` shared call from the platform adapter and restore the previously characterized platform implementation while keeping golden vectors and common tests as regression evidence. For Android runtime planning slices, rollback is to stop consulting shared planning code in the platform facade and keep BLE transport execution, protobuf builders, filesystem gates, and public error mapping in the platform implementation. For Android release packaging, rollback is to remove `polar-ble-sdk-shared.aar` from example/local distribution only after the SDK AAR no longer references `com.polar.shared` classes or after publication metadata supplies the shared dependency. For iOS CocoaPods/Xcode slices, rollback is to remove the `PolarIosSharedBridge` call from the Swift facade and rely on the existing `#if canImport(PolarBleSdkShared)` fallback, then remove the framework search path/script phase only when no production Swift file imports or calls the bridge. For SwiftPM/watchOS, rollback is to delete the generated local-output XCFramework or remove the remote `binaryTarget` URL/checksum entry so `Package.swift` falls back to source-only Swift behavior.

## Review Rules

Do not mix artifact wiring with unrelated parser or runtime migrations. A migration slice that adds Android or iOS consumption must name the delegated behavior family, the Android and iOS tests that prove facade compatibility, the common tests that fail without shared implementation, and the rollback path for every shared-module adoption step. Generated API docs under `docs/polar-sdk-android` and `docs/polar-sdk-ios` remain untouched unless public API documentation is intentionally regenerated for release readiness.
