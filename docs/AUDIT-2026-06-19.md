# Polar BLE SDK — Deep Code Audit (2026-06-19)

Multi-agent audit of the core SDK (Android/Kotlin + iOS/Swift + shared KMP); example apps excluded. 156 findings confirmed by adversarial re-verification against source; 37 candidate findings rejected. Severities below are post-verification (adjusted).

## Executive summary

A 156-finding audit of the Polar BLE SDK (678 source files across Android/Kotlin and iOS/Swift) reveals five cross-cutting failure patterns that account for virtually every finding. The most severe cluster — unguarded byte-buffer access on BLE wire data — appears consistently across both platforms and is directly exploitable by any peripheral, real or spoofed, that sends a malformed notification. The second-most-severe cluster covers cross-thread data races on shared mutable state; there is exactly one @Volatile annotation in the entire Android codebase, while iOS has zero NSLock protections on several callback-written fields. A third cluster of blocking primitives inside coroutine/async contexts (Object.wait, Thread.sleep, synchronized) threatens the stability of the Dispatchers.IO pool on Android and the baseSerialDispatchQueue on iOS. Two correctness clusters complete the picture: operator-precedence and sign-extension bugs that silently corrupt every delta-compressed sensor sample, and a pervasive SDK-wide pattern of force-unwraps and unchecked enum indexing on device-supplied data. An iOS-specific parity gap means several bugs repaired on Android (safe enum lookup, bounds-checked parsers) were never ported to the Swift mirror.

## Cross-cutting themes (ranked)

### [CRITICAL] Unguarded byte-buffer reads on untrusted BLE wire data

Every BLE packet parser in both the Android and iOS codebases indexes into raw byte arrays or Data objects before verifying minimum length. Confirmed crash sites span 14 files: PmdDataFrame.kt/swift (data[9] accessed unconditionally, 10-byte minimum never checked), BleHrClient.kt/swift (data[offset+1] read inside a while(offset < len) guard that passes for odd-length RR sections), BleRscClient.kt/swift (flags-conditional fields read without post-flag-check size validation), BlePsdClient.kt (PsdResponse indexes PsdMessage.entries[data[1].toInt()] raw with no bounds check on the entries array), BlePfcClient.kt/swift (PfcFeature reads data[1] with no guard), PpiData.kt (copyOfRange with no remaining-byte check), EcgData.kt (22-bit value assembled without sign extension), PmdOfflineTrigger.kt (signed Byte used as length, negative-length sliceArray), PmdControlPointResponse.swift (force-unwrap in hex conversion on BLE callback data), BleHtsClient.swift / PolarGattHtsCodec.kt (5-byte minimum never asserted), BleBasClient.swift (data[1] after count > 0 guard), BlePsdClient.swift (PPData reads up to data[4] with no guard), BleAdvertisementContent.swift (manufacturer data parser increments offset past end), PolarUserDeviceSettings.swift / PolarStoredData.swift (allCases[rawValue] subscript with device-supplied index). A malicious or merely buggy peripheral advertising a single malformed packet can crash the host app via AIOOBE/index-out-of-bounds in a BLE callback — no user interaction required.

**Affected:** Android: PmdDataFrame.kt, BlePMDClient.kt, BleHrClient.kt, BleRscClient.kt, BlePsdClient.kt, BlePfcClient.kt, BleGattBase.kt, PmdOfflineTrigger.kt, PpiData.kt, PmdMeasurementType.kt, PolarGattHtsCodec.kt, BleAdvertisementContent.kt, ConnectionHandler.kt. iOS: PmdDataFrame.swift, PmdControlPointResponse.swift, BleHrClient.swift, BleHtsClient.swift, BleRscClient.swift, BlePfcClient.swift, BlePsdClient.swift, BleBasClient.swift, BleAdvertisementContent.swift, PolarUserDeviceSettings.swift, PolarStoredData.swift. Shared: PolarSensorDataParsers.kt

### [CRITICAL] Data races on shared mutable state across BLE callback, IO, and caller threads

The Android codebase has exactly one @Volatile annotation in 678 source files, yet dozens of fields are written from Android Binder/GATT callback threads and read from coroutine/IO threads. Confirmed races: (1) BleGattBase.kt — three plain HashMap fields (characteristics, characteristicsRead, mandatoryNotificationCharacteristics) mutated without locks from callback and caller threads, producing ConcurrentModificationException risk; (2) BleAdvertisementContent.kt — public @JvmField HashMap fields advertisementData/advertisementDataAll cleared and repopulated in the BLE scan callback while being read from subscriber threads; (3) BlePMDClient.kt — currentSettings (HashMap) written on Dispatchers.IO and read on callback thread, previousTimeStampMap likewise; (4) BDDeviceSessionImpl.kt — indicatesPairingProblem, serviceDiscovery Job, and the ATT-queue TOCTOU race across GATT callbacks and coroutines; (5) BDDeviceListenerImpl.kt, GattCallback.kt — indicatesPairingProblem written from broadcast-receiver and coroutine threads; (6) BDBleApiImpl.kt — five plain mutableMapOf() maps (connectSubscriptions, lastDerivedMethodsCache, deviceDataMonitorJob, deviceAvailableFeaturesJob, stopPmdStreamingJob) concurrently mutated without locks, plus a singleton getInstance() with a check-then-act race on a plain var. On iOS: CBDeviceListenerImpl.swift has a TOCTOU race on AtomicList.items[] in sendNextAttNotify; BlePmdClient.swift has an unsynchronized var features written on the BLE callback queue and read from async Tasks; BleBasClient.swift's cachedChargeState/cachedPowerSourcesState have no lock; BlePsFtpClient.swift's progressCallback is written and read on different queues; AtomicInteger.swift's incrementAndGet releases the lock before calling get(), returning a potentially stale value.

**Affected:** Android: BleGattBase.kt, BleAdvertisementContent.kt, BlePMDClient.kt, BDDeviceSessionImpl.kt, BDDeviceListenerImpl.kt, GattCallback.kt, BleDisClient.kt, BleBattClient.kt, BlePfcClient.kt, BDBleApiImpl.kt, PolarSensorSetting.kt. iOS: CBDeviceListenerImpl.swift, CBDeviceSessionImpl.swift, BlePmdClient.swift, BleBasClient.swift, BlePsFtpClient.swift, PolarBleApiImpl.swift, AtomicInteger.swift

### [HIGH] Blocking primitives (Object.wait, Thread.sleep, synchronized) inside coroutine/async dispatch contexts

The SDK systematically uses Java/Objective-C blocking synchronization inside Kotlin coroutines and Swift async contexts, pinning dispatcher threads for seconds to minutes. Five confirmed sites: (1) BleGattBase.kt lines 310-323, 344-346 — Object.wait() inside withContext(Dispatchers.IO), with no while-loop guard against spurious wakeups; (2) BlePsFtpClient.kt lines 226-281 — synchronized(pftpOperationMutex) holding one Dispatchers.IO thread for up to 900 seconds during PFTP operations; under concurrent callers this exhausts the IO pool entirely; (3) BlePfcClient.kt lines 207-241 — synchronized(pfcMutex) + blocking 30-second queue.poll() on Dispatchers.IO; (4) PolarServiceClientUtils.kt lines 155-167 — Thread.sleep(100) busy-spin for up to 10 seconds inside a suspend function, ignoring coroutine cancellation; (5) BlePMDClient.kt line 303-313 — Object.wait() in a bare if-block, vulnerable to spurious wakeup. On iOS: BleH7SettingsClient.swift line 60-66 — inputQueue.poll(30) blocks baseSerialDispatchQueue for 30 seconds; BlePsFtpClient.swift lines 87-90 — waitUntilAllOperationsAreFinished after cancelAllOperations can stall for 90 seconds if operations are blocked in condition waits; PolarServiceClientUtils.swift lines 45-53 — Thread.sleep spin-wait on calling thread. All these patterns make the BLE stack fragile under concurrent usage and unresponsive to cancellation.

**Affected:** Android: BleGattBase.kt, BlePsFtpClient.kt, BlePfcClient.kt, BlePsdClient.kt, BlePMDClient.kt, PolarServiceClientUtils.kt. iOS: BleH7SettingsClient.swift, BlePsFtpClient.swift, PolarServiceClientUtils.swift

### [HIGH] Operator-precedence and sign-extension bugs silently corrupting sensor sample values

Two distinct but related arithmetic bugs corrupt decoded sensor data for the delta-compressed PMD stream, which is the primary data path for ECG, PPG, ACC, gyroscope, and magnetometer on all Polar devices. (1) BlePMDClient.kt line 559: val mask = Int.MAX_VALUE shl bitWidth - 1 — Kotlin's operator precedence evaluates this as (Int.MAX_VALUE) shl (bitWidth - 1). Int.MAX_VALUE is 0x7FFFFFFF; left-shifting it always drops the MSB, producing a mask that never sets the sign bit correctly. Every delta-compressed sensor value that should be negative is instead decoded as a large positive integer. The correct expression is (-1 shl bitWidth). (2) PolarSensorDataParsers.kt line 763: val signMask = Int.MAX_VALUE shl (bitWidth - 1) — the same error in the shared KMP layer, with the additional degenerate case at bitWidth=1 where the result is 0x7FFFFFFF instead of 0xFFFFFFFF. (3) EcgData.kt lines 132-133: the 22-bit ECG TYPE_2 microVolts value is assembled and masked with and 0x3FFFFF but never sign-extended, so all negative ECG voltages arrive as values up to +4,194,303. (4) PolarGattHtsCodec.kt lines 20-22: the IEEE 11073 MEDFLOAT32 mantissa is treated as unsigned 24 bits; temperatures below zero are decoded as large positive numbers. These are silent data integrity bugs that affect every user of the SDK's sensor streaming APIs; they produce no exceptions and are not caught by any existing test.

**Affected:** Android: BlePMDClient.kt (parseDeltaFrame), EcgData.kt (TYPE_2 ECG parser). Shared KMP: PolarSensorDataParsers.kt (parseDeltaFrame), PolarGattHtsCodec.kt (IEEE 11073 mantissa). Downstream: all callers of startStreaming for ECG, PPG, ACC, GYR, MAG, temperature

### [HIGH] Force-unwraps and unsafe enum indexing on device-supplied protobuf and BLE data (iOS/Android parity gap)

A broad class of force-unwraps and unchecked conversions on device-originated data spans the public model layer, with the iOS side significantly worse than Android — many Android bugs have safe guards that were never ported to the Swift equivalents. Representative high-severity sites: (1) PolarSleepData.kt line 90: SleepWakeState.from()!! crashes on any unknown firmware enum value; (2) PolarBleApiImpl.swift lines 2954-2989: force-unwrap of optional until: Date? parameter in deleteStoredDeviceData for six data types; (3) PolarBleApiImpl.swift lines 4039-4066: samples[0] accessed unconditionally on Temperature, Pressure, and SkinTemperature data frames; (4) PolarSensorSetting.swift line 57: value.first! on a Set<UInt32> that may be empty from the wire proto; (5) PolarDataUtils.swift: fatalError for any unrecognized shared measurement type name from firmware, turning a versioning mismatch into a process crash; (6) PolarUserDeviceSettings.swift / PolarStoredData.swift: DeviceLocation.allCases[pbDeviceSettings.rawValue] with no bounds check on a device-supplied integer; (7) PolarAutomaticSamplesUtils.swift lines 65, 101: calendar.date(from:)! force-unwrapped on proto-supplied year/month/day; (8) PolarFirstTimeUseConfig.swift: assert() used for range validation — a no-op in release builds, so invalid height/weight/HR values are silently sent to the device. The iOS public Codable model types (PolarSleepData, Polar247PPiSamplesData, PolarSkinTemperatureData) also declare semantically required stored properties as implicitly-unwrapped optionals, which crash on partial JSON/proto decode rather than propagating nil cleanly.

**Affected:** Android: PolarSleepData.kt, Polar247HrSamplesData.kt, Polar247PPiSamplesData.kt, PolarFirstTimeUseConfig.kt, BDBleApiImpl.kt. iOS: PolarBleApiImpl.swift, PolarSensorSetting.swift, PolarDataUtils.swift, PolarUserDeviceSettings.swift, PolarStoredData.swift, PolarAutomaticSamplesUtils.swift, PolarFirstTimeUseConfig.swift, LogConfig.swift, PolarSleepData.swift, Polar247PPiSamplesData.swift, PolarSkinTemperatureData.swift

### [MEDIUM] Coroutine and resource leaks on disconnect/reconnect cycles

Several coroutine Jobs and Combine subscriptions accumulate on each connect/disconnect cycle and are never cancelled or removed, causing retained closures, orphaned BLE event handlers, and growing memory usage in long-running applications. Key sites: (1) BDBleApiImpl.kt line 3480: an HR-notification coroutine launched in setupDevice is never stored in deviceDataMonitorJob and therefore survives tearDownDevice; it continues dispatching HR events even after the device disconnects. (2) BDBleApiImpl.kt line 192: stopPmdStreamingJob map entries are never removed in tearDownDevice; each reconnect appends a new Job without cancelling the previous. (3) CBDeviceListenerImpl.swift lines 526-546: each completeSessionClose call accumulates a new AnyCancellable in the cancellables Set that is never cleared; over many reconnects this Set grows without bound, retaining closures that hold strong references to self. (4) BlePsFtpClient.swift lines 549-557: the shared wait-notification Flow is not nulled out in reset(), so after reconnect all callers receive a stale pre-disconnect Flow. (5) PolarBleApiImpl.swift line 3142-3147: waitForConnection loops forever with no timeout — if the device never reconnects, the Task leaks indefinitely.

**Affected:** Android: BDBleApiImpl.kt (setupDevice, tearDownDevice, stopPmdStreaming). iOS: CBDeviceListenerImpl.swift (completeSessionClose), BlePsFtpClient.swift (reset, waitForNotification), PolarBleApiImpl.swift (waitForConnection)

### [MEDIUM] Silent error swallowing and missing error propagation in data-read paths

Multiple utility functions catch all Throwable or Error and return empty/null results without logging or re-throwing, making it impossible to distinguish a device with no data from a BLE disconnect mid-transfer, a protobuf parse failure, or an SDK bug. Key sites: (1) PolarSleepUtils.kt lines 75/105: catch(_: Throwable) with zero logging in both readSleepData and readSleepSkinTemperatureResult — the sleep result fields silently remain nil; (2) BDBleApiImpl.kt lines 2726/2783: deleteStoredDeviceData and deleteTelemetryData swallow all exceptions including IndexOutOfBoundsException from line 2684, returning success to the caller when the delete may have failed; (3) PolarAutomaticSamplesUtils.kt lines 35-64/66-97: BLE request and proto-parse exceptions propagate unhandled to callers who expect only domain exceptions; (4) PolarTestUtils.kt line 81: mapSpo2TestEntry calls parseFrom without any try/catch, propagating InvalidProtocolBufferException; (5) PolarOfflineExerciseV2ApiImpl.kt line 146: the function throws synchronously before entering any flow builder, so .catch operators on the returned Flow do not intercept the exception. The pattern of 106 bare catch-Throwable handlers across Android production code warrants a systematic audit.

**Affected:** Android: PolarSleepUtils.kt, BDBleApiImpl.kt, PolarAutomaticSamplesUtils.kt, PolarTestUtils.kt, PolarOfflineExerciseV2ApiImpl.kt. iOS: PolarBleApiImpl+Logging.swift (getLogConfig PFTP session leak on error path), PolarBleApiImpl+PolarOfflineExerciseV2.swift (CancellationError swallowed via try?)

### [MEDIUM] iOS/Android parity drift in correctness fixes and data logic

Several correctness bugs exist on one platform but not the other, indicating that fixes applied to one codebase were not mirrored. Notable divergences: (1) PressureData.swift lines 39-58: the compressed-type-0 parser builds a pressureSamples list but never assigns it to pressureData.samples — the returned object always has zero samples. The Android equivalent does not have this bug. (2) PressureData.swift line 54: the single pressure channel is read at sample[1] instead of sample[0], causing a guaranteed out-of-bounds crash on any compressed pressure frame; Android reads sample[0] correctly. (3) PpgData.swift line 323: the type-5 operationMode passes offset as size and 0 as offset — swapped arguments; Android is correct. (4) PpgData.swift lines 649/722/811/898: statusBits buffer accumulates across samples (declared outside the per-sample loop) in four compressed PPG parsers; Android correctly scopes the buffer per sample. (5) BDBleApiImpl.kt/PolarBleApiImpl.swift lines 1120-1128/1511: lastTimestamp is declared as val/let = 0 and never updated across sub-recording iterations on both platforms — a shared bug where timestamps are not stitched between sub-recordings. (6) Date.localDate() in PolarBleApiImpl.swift applies the current instant's DST offset to historical dates, producing wrong times across DST boundaries.

**Affected:** iOS: PressureData.swift (critical data-loss bugs), PpgData.swift (type-5 offset/size swap, type-7/8/10/13 statusBits accumulation), PolarBleApiImpl.swift (lastTimestamp not chained, DST-skew in localDate()). Both platforms: BDBleApiImpl.kt / PolarBleApiImpl.swift (lastTimestamp not updated in sub-recording loop)

## Top remediation priorities

1. Fix the parseDeltaFrame sign-extension mask on both platforms immediately, as it silently corrupts every ACC/PPG/ECG/GYR/MAG sample decoded from a delta-compressed frame. On Android: change `val mask = Int.MAX_VALUE shl bitWidth - 1` to `val mask = -1 shl bitWidth` in BlePMDClient.kt line 559, and change `val signMask = Int.MAX_VALUE shl (bitWidth - 1)` to `val signMask = -1 shl (bitWidth - 1)` in PolarSensorDataParsers.kt line 763. Add unit tests asserting that a 1-bit delta value of 0b1 decodes to -1.

2. Add a minimum-size guard to every BLE packet parser on both platforms before the first subscript access. The highest-priority sites are: PmdDataFrame.kt/swift (`require(data.size >= 10)`), BleHrClient.kt/swift (change RR loop to `while (offset + 1 < len)`), BlePsdClient.kt (`PsdMessage.entries.getOrNull(data[1].toInt()) ?: return`), PmdControlPointResponse.swift (guard `data.count >= 3`), BleRscClient.kt/swift, BlePfcClient.kt/swift. Fuzz these parsers with truncated and random-length payloads as a regression gate.

3. Replace Object.wait()/synchronized and Thread.sleep with coroutine-native primitives throughout the Android BLE stack. Priority order: (1) BlePsFtpClient.kt — replace `synchronized(pftpOperationMutex)` with `kotlinx.coroutines.sync.Mutex.withLock` so the 900-second PFTP session does not pin a Dispatchers.IO thread; (2) BleGattBase.kt — replace the synchronized/wait/notifyAll pattern in waitServiceDiscovered and waitNotificationEnabled with MutableStateFlow; (3) PolarServiceClientUtils.kt — replace the Thread.sleep(100) busy-spin with `delay(100)` and add `currentCoroutineContext().ensureActive()` per iteration. On iOS, replace the inputQueue.poll(30) blocking call in BleH7SettingsClient.swift with a continuation resolved in processServiceData.

4. Audit and eliminate all unprotected shared mutable state accessed from BLE callback threads. Immediate actions: (1) Annotate indicatesPairingProblem in BDDeviceSessionImpl.kt, BDDeviceListenerImpl.kt, and GattCallback.kt with @Volatile — there is currently exactly one @Volatile in the entire Android codebase; (2) Replace the three plain HashMap fields in BleGattBase.kt with ConcurrentHashMap; (3) Replace advertisementData and advertisementDataAll in BleAdvertisementContent.kt with ConcurrentHashMap and remove @JvmField exposure; (4) Replace currentSettings and previousTimeStampMap in BlePMDClient.kt with ConcurrentHashMap; (5) On iOS, protect BlePmdClient.features and BleBasClient.cachedChargeState/cachedPowerSourcesState with baseSerialDispatchQueue.

5. Fix the two critical iOS PressureData.swift bugs that cause zero samples on every compressed pressure frame: (1) add `pressureData.samples = pressureSamples` before `return pressureData` on line 58; (2) change `Float(sample[1])` to `Float(sample[0])` on line 54. Then audit all other iOS PMD model parsers for the statusBits-accumulation bug (PpgData.swift types 7/8/10/13 — move `var statusBits = [Int8]()` inside the per-sample loop) and the PpgData type-5 offset/size swap (line 323). These are data-loss and crash bugs with no Android counterpart, suggesting the iOS parsers were never systematically reviewed against the Android reference implementation.

6. Replace all force-unwrap and unchecked allCases[] indexing on device-supplied data in the public iOS API layer. The crash-on-forward-compatible-firmware sites are: PolarSensorSetting.swift line 57 (`value.first!` → `value.first ?? 0`), PolarDataUtils.swift (`fatalError` → `return .unknown_type` + log), PolarUserDeviceSettings.swift/PolarStoredData.swift (`allCases[rawValue]` → bounds-checked lookup), PolarAutomaticSamplesUtils.swift (`calendar.date(from:)!` → guard let + continue), PolarBleApiImpl.swift lines 2954-2989 (guard let until before force-unwrap), PolarBleApiImpl.swift lines 4039-4066 (guard !samples.isEmpty). On Android: replace the SleepWakeState.from()!! in PolarSleepData.kt with `?: SleepWakeState.UNKNOWN`.

7. Fix the lastTimestamp stitching bug on both platforms: in BDBleApiImpl.kt line 1120, change `val lastTimestamp = 0uL` to `var lastTimestamp = 0uL` and update it at the end of each sub-recording iteration; mirror the same fix in PolarBleApiImpl.swift line 1511. Add a coroutine cancellation timeout to PolarBleApiImpl.swift waitForConnection (currently an infinite loop), and store the HR-notification coroutine from BDBleApiImpl.kt setupDevice in deviceDataMonitorJob so tearDownDevice cancels it on disconnect.

8. Replace assert() with throwing validation in PolarFirstTimeUseConfig.swift and fix the `getDevicesMatchingConnectionStates` bitwise-OR bug in BDDeviceListenerImpl.kt line 119. The iOS assert() calls are silently disabled in release builds, so invalid height=0 or maxHeartRate=500 values are sent to the device unchecked. Change every assert() to `guard condition else { throw PolarErrors.invalidArgument(...) }`. The Android bitwise-OR bug (`STATE_CONNECTED or STATE_CONNECTING` producing [3] instead of [STATE_CONNECTED, STATE_CONNECTING]) means fetchKnownDevices always returns an empty list, silently breaking auto-reconnect to previously paired devices.

## All confirmed findings

### Severity: high

- **No minimum-size guard before indexing BLE packet bytes 0 and 9** (`security`)
  - sources/Android/android-communications/library/src/main/java/com/polar/androidcommunications/api/ble/model/gatt/client/pmd/PmdDataFrame.kt:17-31
  - The constructor accesses data[0] at line 17 and data[9] at line 28 unconditionally. A malformed BLE notification shorter than 10 bytes throws ArrayIndexOutOfBoundsException in the BLE callback thread, crashing the entire GATT notification pipeline.
  - Fix: Add a guard at the top of the init block: require(data.size >= 10) { "PMD data frame too short: ${data.size}" } and propagate the error to the caller instead of crashing.

- **currentSettings MutableMap read from BLE callback thread and written from IO thread without synchronization** (`concurrency`)
  - sources/Android/android-communications/library/src/main/java/com/polar/androidcommunications/api/ble/model/gatt/client/pmd/BlePMDClient.kt:51, 76, 85, 331, 335
  - currentSettings is a plain mutableMapOf() written in startMeasurement() (Dispatchers.IO, line 331) and read in getFactor() and getSampleRate() which are called from processPmdData() on the BLE notification callback thread. HashMap is not thread-safe; concurrent access can produce stale reads or internal corruption.
  - Fix: Replace with ConcurrentHashMap or guard all accesses with a dedicated lock. Since getFactor/getSampleRate are called for every frame, a ConcurrentHashMap is preferable to avoid contention.

- **Object.wait() called without a while-loop guard against spurious wakeups** (`concurrency`)
  - sources/Android/android-communications/library/src/main/java/com/polar/androidcommunications/api/ble/model/gatt/client/pmd/BlePMDClient.kt:303-313
  - readFeature() calls mutexFeature.wait() at line 305 in a bare if-block. A spurious wakeup causes the method to continue with pmdFeatureData still null, falling through to the 'Undefined device error' throw instead of waiting again for the actual notification.
  - Fix: Replace the if (pmdFeatureData == null) { mutexFeature.wait() } pattern with while (pmdFeatureData == null) { mutexFeature.wait() } to handle spurious wakeups correctly.

- **Operator-precedence bug in parseDeltaFrame sign-extension mask produces wrong decoded values** (`correctness`)
  - sources/Android/android-communications/library/src/main/java/com/polar/androidcommunications/api/ble/model/gatt/client/pmd/BlePMDClient.kt:559
  - val mask = Int.MAX_VALUE shl bitWidth - 1 is parsed as Int.MAX_VALUE shl (bitWidth - 1) due to Kotlin operator precedence (subtraction binds tighter than shl). Int.MAX_VALUE is 0x7FFFFFFF; shifting it left drops the MSB, so the mask never has the sign bit set correctly. Every sign-extended delta sample (ACC, PPG, ECG, gyro, magnetometer) is corrupted for any bitWidth other than 1.
  - Fix: Add explicit parentheses: val mask = (Int.MAX_VALUE shl bitWidth) - 1, or equivalently val mask = -1 shl bitWidth to produce a proper two's-complement sign-extension mask.

- **Signed Byte used as settings-length field causes negative-length sliceArray crash on malformed packets** (`security`)
  - sources/Android/android-communications/library/src/main/java/com/polar/androidcommunications/api/ble/model/gatt/client/pmd/PmdOfflineTrigger.kt:58-66
  - triggerSettingsLength is read as a signed Kotlin Byte (line 58). When a peripheral or attacker sends a length byte with bit-7 set (values 128–255), Kotlin sign-extends it to a negative Int. sliceArray(offset until (offset + negativeLength)) produces an empty or reversed range that throws IllegalArgumentException, crashing the BLE notification handler.
  - Fix: Read the length as unsigned: val triggerSettingsLength = bytes[offset].toInt() and 0xFF, then validate that offset + triggerSettingsLength <= bytes.size before slicing.

- **Unsynchronized plain HashMap fields mutated and read from different threads** (`concurrency`)
  - sources/Android/android-communications/library/src/main/java/com/polar/androidcommunications/api/ble/model/gatt/BleGattBase.kt:26-32
  - Three plain `HashMap` fields (`characteristics`, `characteristicsRead`, `mandatoryNotificationCharacteristics`) are mutated from any caller thread via `addCharacteristic`, `addCharacteristicNotification`, `removeCharacteristicNotification`, and `reset()`, while being read concurrently from BLE callback threads via `containsCharacteristic`, `isAutomatic`, `isAutomaticRead`, `containsNotifyCharacteristic`, `getNotificationAtomicInteger`, and the `reset()` iteration over `mandatoryNotificationCharacteristics.values`. No synchronization guards these maps, creating a data race that can cause `ConcurrentModificationException` or corrupt state silently.
  - Fix: Replace all three `HashMap` fields with `ConcurrentHashMap`, or guard every read and write with a single dedicated lock; `reset()` must hold the same lock when iterating.

- **Object.wait() inside suspend function can permanently block a coroutine dispatcher thread** (`concurrency`)
  - sources/Android/android-communications/library/src/main/java/com/polar/androidcommunications/api/ble/model/gatt/BleGattBase.kt:310-323
  - `waitServiceDiscovered` is a `suspend fun` that calls `withContext(Dispatchers.IO)` and then calls `(serviceDiscovered as Object).wait()` — a raw Java blocking wait — inside a `synchronized` block. This pins the `Dispatchers.IO` thread for the entire duration of the wait (which can be indefinitely long), starving other coroutines sharing that pool. The same pattern exists in `waitNotificationEnabled` (line 344).
  - Fix: Replace the `synchronized`/`wait()`/`notifyAll()` pattern with a `kotlinx.coroutines.flow.MutableStateFlow` (or `Channel`/`Mutex`/`suspendCancellableCoroutine`), which suspends without blocking any thread.

- **Blocking synchronized block called inside suspend function starves Dispatchers.IO** (`concurrency`)
  - sources/Android/android-communications/library/src/main/java/com/polar/androidcommunications/api/ble/model/gatt/client/psftp/BlePsFtpClient.kt:226-281
  - `request()`, `query()`, `write()`, and `sendNotification()` all call `synchronized(pftpOperationMutex) { ... }` — a potentially 90-second or 900-second blocking critical section — from within `withContext(Dispatchers.IO)` or a `channelFlow`. Because `synchronized` blocks thread preemption, a single long-running PFTP operation holds one `Dispatchers.IO` thread captive for up to 15 minutes (extended timeout). Under concurrent callers the entire IO pool can be exhausted, deadlocking all coroutine-based BLE communication.
  - Fix: Replace `synchronized(pftpOperationMutex)` with `kotlinx.coroutines.sync.Mutex` and call it with `mutex.withLock { ... }` so the coroutine suspends instead of blocking the thread.

- **RR-interval parsing reads data[offset+1] without checking offset+1 < data.size, causing AIOOBE on malformed BLE packet** (`correctness`)
  - sources/Android/android-communications/library/src/main/java/com/polar/androidcommunications/api/ble/model/gatt/client/BleHrClient.kt:99-102
  - The `while (offset < len)` loop (line 100) guards only that `offset < len`, but reads `data[offset+1]` inside the loop body (line 101). When the peripheral sends a packet with an odd number of trailing bytes in the RR section (i.e., `data.size - offset == 1`), `offset` equals `len - 1`, the while condition is true, and `data[offset + 1]` is `data[len]`, throwing `ArrayIndexOutOfBoundsException` and crashing the BLE callback.
  - Fix: Change the loop guard to `while (offset + 1 < len)` so the loop only runs when a full 2-byte RR value is available.

- **RSC measurement parsing has no bounds checks; malformed packets cause ArrayIndexOutOfBoundsException** (`correctness`)
  - sources/Android/android-communications/library/src/main/java/com/polar/androidcommunications/api/ble/model/gatt/client/BleRscClient.kt:51-61
  - The parser unconditionally reads `data[index++]` (speed: 2 bytes, cadence: 1 byte at indices 1-3), then conditionally reads 2 more for stride length and 4 more for total distance, all without checking that `data.size` is sufficient. The RSC feature read on line 79 also reads `data[1]` without checking `data.size >= 2`. A peripheral sending a short advertisement can crash the BLE callback thread with `ArrayIndexOutOfBoundsException`.
  - Fix: Add `if (data.size < minimumRequiredSize) return` guards before each field read, where the minimum is derived from the flags byte.
